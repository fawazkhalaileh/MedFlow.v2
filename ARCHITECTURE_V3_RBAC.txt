SECTION 1: PROBLEM DIAGNOSIS
================================================================================

1.1 CURRENT STATE OF THE ROLE EXPERIENCE

The application currently has no meaningful role separation at the UI level.
Every authenticated user, regardless of their employee_type value, is redirected
to the same /dashboard route upon login. The dashboard renders the same sidebar,
the same KPI widgets, and the same action buttons whether the authenticated user
is a system_admin or a secretary.

The User model carries an employee_type field with the values:
  system_admin, branch_manager, secretary, technician, doctor, nurse, finance

This field exists in the database and is populated by the seeder. However, the
application makes no use of it at the routing, middleware, controller, or view
layer. The field is informationally present but functionally inert.

1.2 SPECIFIC RISKS OF THE CURRENT STATE

RISK 1 — DATA EXPOSURE
A secretary can currently navigate to any admin route, view any patient across
any branch, and access employee management if those routes exist. There is no
enforcement preventing cross-branch data access. A secretary at Branch A can
view Branch B patient records simply by manipulating the URL.

RISK 2 — ACCIDENTAL DATA MODIFICATION
Staff who should only view or only create records (technicians, nurses) can
currently edit or delete records they have no business modifying. There is no
create/edit/delete gate other than what Laravel's default middleware provides
(authentication only).

RISK 3 — AUDIT TRAIL GAPS
Because all roles perform actions through the same unguarded routes, the activity
log cannot distinguish between an admin intentionally changing a setting and a
secretary accidentally clicking the same button. This makes compliance reporting
and incident investigation unreliable.

RISK 4 — OPERATIONAL CONFUSION
A secretary's primary job is front-desk intake: booking appointments, registering
patients, and managing the check-in queue. Showing them the company-wide KPI
dashboard, branch management panels, and role permission editors creates cognitive
overload and increases the likelihood of accidental actions.

RISK 5 — PRIVILEGE ESCALATION PATH
If an attacker compromises a low-privilege account (e.g., a secretary), the
absence of server-side role checks means they gain equivalent access to the
application as the system_admin. The only protection is the login form itself.

1.3 TECHNICAL PATTERNS THAT ARE MISSING

MISSING PATTERN 1 — ROLE-CHECK MIDDLEWARE
No middleware class enforces employee_type on route access. There is no
equivalent to a CheckRole or EnsureEmployeeType middleware registered in the
application. Routes are only protected by the 'auth' middleware.

MISSING PATTERN 2 — ROUTE GROUPING BY ROLE
Routes are not grouped by role scope. There is no /admin/* group restricted to
system_admin, no /secretary/* group, and no /clinical/* group. All routes exist
in a flat namespace reachable by any authenticated user.

MISSING PATTERN 3 — BRANCH SCOPING IN CONTROLLERS
Controllers do not filter queries by the authenticated user's primary_branch_id.
A query such as Patient::all() returns every patient in the database regardless
of which branch the requesting user belongs to. Branch-scoped queries must be
applied as a base filter for non-admin roles.

MISSING PATTERN 4 — ROLE-AWARE VIEWS
The Blade views use no role conditionals. The sidebar is a static partial
rendered identically for all users. There are no @if(auth()->user()->employee_type
=== 'secretary') guards in any view. View composers are not registered to inject
role context.

MISSING PATTERN 5 — POST-LOGIN REDIRECT BY ROLE
The AuthenticatedSessionController redirects all users to /dashboard. There is
no post-login branching that sends a secretary to /secretary/dashboard or an
admin to /admin/dashboard based on employee_type.

MISSING PATTERN 6 — PERMISSION MODEL ENFORCEMENT
The Role and Permission models exist in the database but are not applied to gate
checks, policy classes, or route middleware. The permission seeder runs but the
permissions are never consulted at runtime.

SECTION 2: ROLE REDESIGN SUMMARY
================================================================================

2.1 ROLE OVERVIEW TABLE

The following table defines each role's routing, purpose, scope, and edit access.

ROLE             | LANDING PAGE               | PRIMARY PURPOSE               | BRANCH SCOPE         | EDIT ACCESS
-----------------|----------------------------|-------------------------------|----------------------|---------------------------
system_admin     | /admin/dashboard           | System governance, all config | All branches         | Full — all modules
branch_manager   | /branch/dashboard          | Branch operations oversight   | Own branch only      | Staff schedules, reports
secretary        | /secretary/dashboard       | Front-desk, booking, intake   | Own branch only      | Appointments, patients (create/edit)
technician       | /clinical/queue            | Session execution             | Own branch only      | Session notes only
doctor           | /clinical/review           | Medical review, plans         | Own branch only      | Treatment plans, notes
nurse            | /clinical/queue            | Session support               | Own branch only      | Session notes only
finance          | /finance/dashboard         | Billing, revenue reporting    | Own branch only      | Billing records only

2.2 ROLE HIERARCHY NOTES

system_admin is the only role that can:
  - Create, edit, or delete branches
  - Create, edit, or delete user accounts (employees)
  - Manage roles and permissions
  - View activity logs
  - Access system settings
  - View data across all branches simultaneously

branch_manager can:
  - View all patients and appointments within their branch
  - Edit staff schedules within their branch
  - View branch-level reports
  - Cannot create or delete employees
  - Cannot access other branches

secretary can:
  - Create and edit patient records
  - Create, edit, and cancel appointments
  - Move appointment cards through the front-desk Kanban stages
  - Cannot access billing records
  - Cannot view treatment details or clinical notes

technician and nurse can:
  - View their assigned appointment queue
  - Add session notes to their assigned sessions
  - Mark sessions as complete
  - Cannot view other staff's queues
  - Cannot create or cancel appointments

doctor can:
  - View all appointments and patients within their branch
  - Create and edit treatment plans
  - Write clinical notes
  - Review session records
  - Cannot access billing or admin panels

finance can:
  - View billing records and payment status for all branch patients
  - Generate revenue reports
  - Cannot view clinical notes or treatment details
  - Cannot create or edit appointments

SECTION 3: SECRETARY WORKSPACE DESIGN
================================================================================

3.1 LANDING PAGE: /secretary/dashboard

The secretary dashboard is designed for speed. Everything the secretary needs for
a typical workday must be visible above the fold without scrolling.

ABOVE THE FOLD LAYOUT (three columns):

LEFT COLUMN — TODAY'S QUEUE SUMMARY
  - Total appointments today (count badge)
  - Breakdown by status: Booked, Confirmed, Arrived, Checked In, Waiting
  - Alert count: No-shows from yesterday (requires follow-up)
  - Quick link: "View Full Kanban" button

CENTER COLUMN — QUICK ACTIONS
  - Primary action button: "+ New Appointment" (routes to /appointments/create)
  - Secondary button: "Register New Patient" (routes to /patients/create)
  - Search bar: "Search Patient" — live search by name, phone, email, patient_code
    (AJAX to /patients/search endpoint, results appear as dropdown below field)
  - Upcoming reminder: next 3 appointments within 1 hour with patient name, time,
    service name

RIGHT COLUMN — BRANCH STATS (TODAY)
  - Appointments scheduled vs completed
  - Walk-in count (appointments with source = 'walk_in')
  - Cancellations today
  - Available time slots remaining (computed from staff schedules)

BELOW THE FOLD:
  - Compact Kanban view showing Booked, Confirmed, Arrived, Checked In columns
    (the four columns the secretary acts on most)
  - Recent patient registrations (last 5, with edit links)

3.2 BRANCH SCOPING RULES FOR SECRETARY

All data displayed to a secretary is filtered by their primary_branch_id. This
is enforced at the controller level, not at the view level.

Rule 1: Patient query base — Patient::where('branch_id', $user->primary_branch_id)
Rule 2: Appointment query base — Appointment::where('branch_id', $user->primary_branch_id)
Rule 3: Staff list for assignment — User::where('primary_branch_id', $user->primary_branch_id)
Rule 4: Service list — Service::where('branch_id', $user->primary_branch_id)
           OR Service::whereNull('branch_id') for company-wide services

No query in any controller serving secretary routes should return records outside
the secretary's primary_branch_id. A global scope or a trait (BranchScoped) can
enforce this automatically if applied to the Eloquent model.

3.3 WHAT SECRETARY CAN SEE, EDIT, AND NOT ACCESS

CAN VIEW:
  - All patients registered to their branch
  - All appointments for their branch (all statuses)
  - Patient contact information
  - Appointment history per patient (dates, services, status)
  - Staff list (name and role only — no salary or HR data)
  - Available time slots per staff member
  - Services offered at their branch
  - Basic lead records (name, contact, source) for their branch

CAN CREATE:
  - New patient records
  - New appointments
  - Follow-up records attached to appointments
  - Notes of type 'reception' on appointments

CAN EDIT:
  - Patient contact information (name, phone, email, address)
  - Appointment status (within their allowed Kanban transitions)
  - Appointment date, time, service, assigned staff (before session starts)
  - Cancellation reason on cancelled appointments

CANNOT ACCESS:
  - Clinical notes (doctor/nurse notes)
  - Treatment plans
  - Treatment session clinical data
  - Billing records or payment information
  - Employee management
  - Branch management
  - Role or permission management
  - System settings
  - Activity logs
  - Any data from other branches
  - Patient medical history in clinical detail

3.4 KANBAN COLUMN TRANSITIONS ALLOWED BY SECRETARY

Secretary can move appointment cards through these transitions:
  Booked -> Confirmed
  Confirmed -> Arrived
  Arrived -> Checked In
  Checked In -> Waiting
  Booked -> Cancelled (with required cancellation reason)
  Confirmed -> Cancelled (with required cancellation reason)
  Any status -> No Show (for same-day past appointments only)

Secretary CANNOT move cards to:
  Ready for Technician (technician or branch manager only)
  In Session (technician only)
  Completed (technician or doctor only)
  Follow-up Needed (doctor only)

SECTION 4: SUPER ADMIN WORKSPACE DESIGN
================================================================================

4.1 LANDING PAGE: /admin/dashboard

The Super Admin dashboard presents company-wide operational intelligence. It is
designed for executive oversight, not day-to-day clinical operations.

ABOVE THE FOLD — COMPANY-WIDE KPI STRIP:
  - Total active patients (all branches)
  - Total appointments today (all branches)
  - Revenue this month (all branches combined)
  - Total employees (active)
  - Total branches (active)
  - System alerts (failed jobs, error logs — count badge)

BRANCH PERFORMANCE GRID (one card per branch):
  Each branch card shows:
  - Branch name and location
  - Appointments today / this week
  - Patients registered this month
  - Staff on duty today
  - Revenue this month (branch-level)
  - Quick link to branch detail page

ADMIN GOVERNANCE PANEL (below branch grid):
  - Recent activity log entries (last 10 actions, with user, action, timestamp)
  - Pending employee account requests (if any)
  - New lead registrations across all branches (last 24 hours)
  - System health: database size, storage used, queue status

4.2 WHAT ONLY SUPER ADMIN CAN DO

Super Admin exclusive capabilities:
  1. Create, edit, deactivate, delete Branch records
  2. Create employee accounts and assign employee_type
  3. Reset any employee's password
  4. Assign roles to employees
  5. Create, edit, delete Role and Permission records
  6. View full activity_logs for all users across all branches
  7. Access /admin/settings for system configuration
  8. View cross-branch reports and comparisons
  9. Export data (patients, appointments, revenue) at company level
  10. Manage the Service catalog globally
  11. Impersonate other users for debugging (if implemented)
  12. Purge or archive records

4.3 ADMIN SECTION PAGES

The /admin prefix contains the following pages, all behind 'role:system_admin':

  /admin/dashboard        — Company overview (described above)
  /admin/branches         — Branch list, create, edit, deactivate
  /admin/branches/{id}    — Branch detail with staff list and metrics
  /admin/employees        — All employees across all branches
  /admin/employees/create — Create new employee account
  /admin/employees/{id}   — Edit employee: name, email, type, branch, status
  /admin/roles            — Role list
  /admin/roles/{id}       — Role edit with permission checkboxes
  /admin/permissions      — Permission list, create, edit
  /admin/activity-logs    — Full audit trail, filterable by user/date/action
  /admin/settings         — System configuration: clinic name, timezone, currency

================================================================================

5.1 LANDING PAGE: /branch/dashboard

The branch manager sees an operations board scoped entirely to their branch.
The primary_branch_id of the branch manager determines all data shown.

ABOVE THE FOLD — TODAY'S OPERATIONS SUMMARY:
  - Appointments today: total, completed, pending
  - Staff on duty: list of employees scheduled today with their appointment counts
  - Revenue today: sum of completed session billings
  - Cancellation rate this week (percentage)

OPERATIONS BOARD (below the fold):
  - Mini Kanban showing all appointment statuses for today
  - Staff utilization bar: each technician/doctor and their session load
  - No-show tracker: appointments marked No Show in last 7 days
  - Follow-up required: appointments with status follow_up_needed

5.2 STAFF VISIBILITY

The branch manager can see:
  - Full name and employee_type of all staff in their branch
  - Each staff member's appointment schedule for the day
  - Session counts per staff member per week
  - Staff availability (based on schedule records if implemented)

The branch manager CANNOT see:
  - Staff salary or HR compensation data
  - Staff accounts in other branches
  - Ability to create or delete employee accounts
  - Password reset for employees

5.3 WHAT BRANCH MANAGER CANNOT DO

Branch managers are operationally powerful within their branch but are not
system administrators. The following actions are blocked:

  - Cannot access /admin/* routes
  - Cannot create or delete employee accounts
  - Cannot manage roles or permissions
  - Cannot view other branches' data
  - Cannot change system settings
  - Cannot view company-wide revenue or patient totals
  - Cannot access activity logs (only admin can)
  - Cannot delete branches or modify branch configuration
  - Cannot delete patient records

5.4 BRANCH MANAGER AVAILABLE ROUTES

  /branch/staff                   — Staff list for branch (view only)
  /branch/reports                 — Branch-level reports: revenue, appointments
  /branch/reports/revenue         — Revenue breakdown by service and period
  /branch/reports/appointments    — Appointment volume and outcome stats

  - Create and edit treatment plans
  - Write clinical notes on any patient in their branch
  - Move appointment cards to Follow-up Needed and Completed

CANNOT DO:
  - Create or cancel appointments (doctors are clinical, not front-desk)
  - Edit patient contact information
  - Access billing records
  - Access /admin routes
  - View other branches

6.5 NURSE WORKSPACE

Nurses share the /clinical/queue landing page with technicians. Their permission
set is identical to technicians in this MVP. The only distinction is their
employee_type value ('nurse' vs 'technician'), which is displayed on their profile
and used for filtering the staff assignment dropdown in the booking form.

SECTION 8: ADMIN SECTION DESIGN

8.1 ROUTE PREFIX AND MIDDLEWARE

All admin routes use the /admin prefix and the route name prefix admin.*.
All admin routes are protected by two middleware layers:
  1. 'auth' — must be authenticated
  2. 'role:system_admin' — employee_type must equal 'system_admin'

Any request to /admin/* from a non-system_admin user returns a 403 response
rendered by the resources/views/errors/403.blade.php view. The 403 view should
display a friendly message and a link back to the user's appropriate dashboard.

8.2 ADMIN ROUTES SPECIFICATION

Route::prefix('admin')->name('admin.')->middleware(['auth', 'role:system_admin'])
->group(function () {

    Route::get('/dashboard', [AdminController::class, 'dashboard'])
         ->name('dashboard');

    Route::resource('branches', AdminBranchController::class);
    Route::get('branches/{branch}/staff', [AdminBranchController::class, 'staff'])
         ->name('branches.staff');

    Route::resource('employees', AdminEmployeeController::class);
    Route::post('employees/{employee}/reset-password',
         [AdminEmployeeController::class, 'resetPassword'])
         ->name('employees.reset-password');

    Route::resource('roles', AdminRoleController::class);
    Route::post('roles/{role}/sync-permissions',
         [AdminRoleController::class, 'syncPermissions'])
         ->name('roles.sync-permissions');

    Route::resource('permissions', AdminPermissionController::class);

    Route::get('activity-logs', [AdminActivityLogController::class, 'index'])
         ->name('activity-logs.index');
    Route::get('activity-logs/{log}', [AdminActivityLogController::class, 'show'])
         ->name('activity-logs.show');

    Route::get('settings', [AdminSettingsController::class, 'index'])
         ->name('settings.index');
    Route::put('settings', [AdminSettingsController::class, 'update'])
         ->name('settings.update');
});

9.1 BOOKING FORM URL AND CONTROLLER

URL: /appointments/create
Controller: AppointmentController@create
View: resources/views/appointments/create.blade.php
Route middleware: ['auth', 'role:secretary,branch_manager,system_admin']

9.2 STEP-BY-STEP BOOKING FLOW

The booking form is a single-page multi-step form. Each step is a distinct
visual section. The form does not use separate page loads between steps — steps
are toggled by JavaScript. On submission, the entire form POSTs to
/appointments (AppointmentController@store).

STEP 1 — PATIENT LOOKUP

  Display: Large search bar labeled "Search Patient (name, phone, email, or code)"

  Behavior:
    - User types 2 or more characters
    - JavaScript sends AJAX GET to /patients/search?q={query}
    - Controller returns JSON array of matching patients (max 10 results)
    - Results render below the search bar as a dropdown list
    - Each result row shows: patient_code | full name | phone | date of birth
    - User clicks a result to select the patient
    - On selection: patient card appears showing name, code, phone, date of birth,
      last visit date, total visits, any active treatment plan
    - Hidden input: patient_id is populated with the selected patient's ID

  No Match Found:
    - If the search returns zero results, a "No patient found — Register New"
      button appears below the search bar
    - Clicking this button expands an inline patient registration sub-form
      (see Section 9.3)

  Return Patient Indicator:
    - If selected patient has prior appointments: show "Returning Patient" badge
    - If selected patient has no prior appointments: show "New Patient" badge

STEP 2 — SERVICE SELECTION

  Display: Grid of available services as selectable cards

  Services loaded from: Service::where('branch_id', $user->primary_branch_id)
                         ->orWhereNull('branch_id')->get()

  Each service card shows:
    - Service name
    - Duration (e.g., 45 min)
    - Price
    - Category label (e.g., Laser, Skin, Consultation)

  User clicks a card to select it. Selected card gets a highlight border.
  Multiple services can be selected if the branch supports multi-service booking.
  Hidden input: service_id (or service_ids array) populated on selection.

STEP 3 — DATE AND TIME SELECTION

  Display: Calendar date picker + time slot grid

  Date picker:
    - Default: today's date
    - Available dates: today and future (past dates disabled)
    - On date selection: time slot grid refreshes via AJAX

  Time slot grid:
    - Load available slots from /appointments/available-slots?
      date={date}&staff_id={staff_id}&service_id={service_id}&branch_id={branch_id}
    - Slots shown as 30-minute or 60-minute blocks depending on service duration
    - Taken slots rendered as greyed-out and unclickable
    - Available slots rendered as clickable buttons

  User selects a time slot. Hidden input: appointment_date, appointment_time.

STEP 4 — STAFF ASSIGNMENT

  Display: Dropdown or card grid of available staff

  Staff loaded from: User::where('primary_branch_id', $user->primary_branch_id)
                     ->whereIn('employee_type', ['technician', 'doctor', 'nurse'])
                     ->get()

  Optional: if no specific staff preference, "Any Available" option appears
  which will leave assigned_staff_id null until the branch manager assigns it.

  Staff card shows: name, employee_type label, photo placeholder.

STEP 5 — NOTES AND SOURCE

  Display: Two fields

  - Appointment Notes (textarea, optional): internal front-desk note
  - Booking Source (select): phone, walk_in, website, referral, social_media
  - First Visit checkbox (auto-checked if patient has no prior appointments)

STEP 6 — REVIEW AND CONFIRM

  Display: Summary card showing all selections:
    - Patient: [name] ([patient_code])
    - Service: [service name]
    - Date: [formatted date]
    - Time: [time]
    - Staff: [staff name or "Any Available"]
    - Notes: [notes or "None"]
    - Source: [source]

  Two buttons:
    - "Back" — returns to previous step
    - "Book Appointment" — submits the form

9.3 INLINE PATIENT CREATION SUB-FORM

Triggered when: search returns no results and user clicks "Register New Patient"

Fields revealed inline (no page redirect):
  - First Name (required)
  - Last Name (required)
  - Phone (required)
  - Email (optional)
  - Date of Birth (optional)

10.2 APPOINTMENT CARD DESIGN

    - Service name (truncated)

  THIRD ROW:
    - Staff name or "Unassigned"
    - Status badge (color-coded, matches column color)

  ALERT FLAGS (appear as small icon badges on the card):
    - Red clock icon: appointment is overdue (past scheduled time and not checked in)
    - Orange warning icon: allergy or contraindication note on patient record
    - Yellow flag: patient requested specific staff but staff is not assigned
    - Blue phone icon: appointment was booked via phone (source = 'phone')

  CARD FOOTER:
    - "View" link: opens appointment detail slide-over panel
    - Quick action button: context-sensitive based on current status

10.4 ROLE PERMISSIONS ON KANBAN

CARD MOVEMENT PERMISSIONS (who can drag/move cards to which columns):

  system_admin:

  branch_manager:

10.5 DRAG AND DROP IMPLEMENTATION NOTE

MVP Phase 1 uses button-based status changes (a dropdown or quick action button

such as SortableJS or Dragula. The server-side permission logic for transitions
is identical regardless of whether the UI uses drag-and-drop or buttons.

Status change endpoint:

The controller checks the transition against the role's allowed transitions before
updating the record. Invalid transitions return a 422 with an error message.

13.1 CHECKROLE MIDDLEWARE

Create the file: app/Http/Middleware/CheckRole.php

<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

class CheckRole
{
    public function handle(Request $request, Closure $next, string ...$roles): Response
    {
        if (!$request->user()) {
            return redirect()->route('login');
        }

        $userType = $request->user()->employee_type;

        // Map legacy 'admin' role field to system_admin employee_type
        if ($request->user()->role === 'admin' && empty($userType)) {
            $userType = 'system_admin';
        }

        if (!in_array($userType, $roles)) {
            abort(403, 'Access denied. You do not have permission to view this page.');
        }

        return $next($request);
    }
}

In bootstrap/app.php, within the withMiddleware callback:

$app->withMiddleware(function (Middleware $middleware) {

13.4 BRANCH SCOPING IN CONTROLLERS

Create a trait: app/Traits/BranchScoped.php

<?php

namespace App\Traits;

trait BranchScoped
{
    protected function branchId(): int
    {
        $user = auth()->user();
        if ($user->employee_type === 'system_admin') {
            return request()->input('branch_id', 0); // admin can query any branch
        }
        return (int) $user->primary_branch_id;
    }

    protected function scopeToUserBranch($query)
    {
        $user = auth()->user();
        if ($user->employee_type !== 'system_admin') {
            $query->where('branch_id', $user->primary_branch_id);
        }
        return $query;
    }
}

Usage in AppointmentController:

use App\Traits\BranchScoped;

class AppointmentController extends Controller
{
    use BranchScoped;

    public function index()
    {
        $appointments = $this->scopeToUserBranch(Appointment::query())
            ->with(['patient', 'service', 'staff'])
            ->get();
        return view('appointments.index', compact('appointments'));
    }
}

Usage in PatientController@search:

public function search(Request $request)
{
    $q = $request->input('q', '');
    if (strlen($q) < 2) {
        return response()->json([]);
    }

    $query = Patient::query();

    if (auth()->user()->employee_type !== 'system_admin') {
        $query->where('branch_id', auth()->user()->primary_branch_id);
    }

    $results = $query->where(function ($q2) use ($q) {
            $q2->where('first_name', 'like', "%{$q}%")
               ->orWhere('last_name', 'like', "%{$q}%")
               ->orWhere('phone', 'like', "%{$q}%")
               ->orWhere('email', 'like', "%{$q}%")
               ->orWhere('patient_code', 'like', "%{$q}%");
        })
        ->select(['id', 'patient_code', 'first_name', 'last_name', 'phone', 'email',
                  'date_of_birth'])
        ->limit(10)
        ->get();

    return response()->json($results);
}

13.6 POST-LOGIN REDIRECT BY ROLE

private function redirectBasedOnRole($user): string
{
    return match($user->employee_type) {
        'system_admin'   => route('admin.dashboard'),
        'branch_manager' => route('branch.dashboard'),
        'secretary'      => route('secretary.dashboard'),
        'technician',
        'nurse'          => route('clinical.queue'),
        'doctor'         => route('clinical.review'),
        'finance'        => route('finance.dashboard'),
        default          => route('dashboard'),
    };
}

public function store(LoginRequest $request): RedirectResponse
{
    $request->authenticate();
    $request->session()->regenerate();
    return redirect()->to($this->redirectBasedOnRole($request->user()));
}
